### Growth Rate Computation

**Formula:** Annualized CAGR from OEWS employmentHistory
```
growthRate = (latest_employment / earliest_employment) ^ (1 / years) - 1
years = 2025 - 2019 = 6
```

### Bundle Hygiene — Library Loaders

**Problem:** 
- `data/jolts.json` (~1.1 MB) + `data/warn-notices.json` (~0.5 MB) = 1.6 MB total
- If imported in `lib/data.ts`, every page bundles them (even `/careers`, `/sectors` don't use them)

**Solution implemented:**
- Created `lib/jolts.ts` — imports `jolts.json`, exports loaders (`getJoltsData()`, `getJoltsLayoffsPeak()`, etc.)
  - Imported ONLY by `/pulse` route chunk
- Created `lib/warn.ts` — imports `warn-notices.json`, exports loaders (`getWarnData()`, `getWarnSummary()`, `getWarnNotices()`, etc.)
  - Imported ONLY by `/layoffs` route chunk
- Verified: `/careers`, `/sectors`, `/skills`, `/global` do NOT import these loaders

**Result:** Main bundle routes remain lean; only `/pulse` and `/layoffs` pay the cost (by design)

## Key Lessons Recorded

**Lesson 1: Government data API landscape**
- BLS data (JOLTS, OEWS) = reliable API access
- Company-level WARN notices = no federal API; state-by-state scraping only
- Forward projections (BLS Employment Projections) = inaccessible (403, not archived)
- **For future data work:** Verify API availability early; have fallback plan

**Lesson 2: Bundle hygiene for large snapshots**
- Large route-specific JSON snapshots (>1 MB) belong in dedicated loader modules
- Import only by their route chunks, NOT in shared lib/data.ts
- Verified bundle size impact: 1.6 MB snapshot footprint only on /pulse + /layoffs routes

**Lesson 3: Authentic growth rate derivation**
- When official projections unavailable, derive from historical data (authentic CAGR)
- More honest than synthesis or external estimates
- Disclose methodology (e.g., "based on 2019–2025 trends, not forward projections")
